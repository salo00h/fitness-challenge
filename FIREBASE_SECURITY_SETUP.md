# إعداد الأمان النهائي — Fitness Challenge (fitness-challenge-6061e)

هذا الملف يشرح **ماذا تغيّر ولماذا**، وكيف تكملين الخطوات اليدوية المتبقية في Firebase Console، بدون أي مصطلحات معقدة غير ضرورية.

---

## 1) ما المشكلة السابقة؟

- قاعدة البيانات كانت في **Test Mode**: أي شخص، بدون تسجيل دخول، يستطيع قراءة أو تعديل أو حذف أي شيء في Firestore. هذه الفترة على وشك الانتهاء وتتحول تلقائيًا إلى "امنع كل شيء" — وهو ما كان سيوقف التطبيق بالكامل.
- تسجيل الدخول لم يكن حقيقيًا: الاسم يُقرأ من `localStorage`، وكلمة المرور تُشفّر بـ SHA-256 **داخل المتصفح** وتُقارن هناك أيضًا. لا يوجد أي طرف (خادم أو Firestore rules) يتحقق من هوية المستخدمة فعليًا.
- بسبب ذلك: أي شخص يفتح أدوات المطورة (DevTools) يستطيع تغيير اسمه، قراءة `passwordHash` لأي مشاركة، أو تعديل تقدّم مشاركة أخرى مباشرة.
- حماية `admin.html` كانت كلمة مرور ثابتة (`"1234"`) داخل كود JavaScript يصل لكل متصفح — ليست حماية حقيقية، وأي تعديل Firestore إداري (إضافة تمرين، حذف مشاركة...) لم يكن مرتبطًا بها أصلًا.

## 2) ما الحل الذي تم اختياره؟ (مقارنة الخيارات A–F)

| الخيار | التقييم |
|---|---|
| **A. تمديد Test Mode** | مرفوض صراحة من الطلب، وهو تأجيل للمشكلة فقط دون حلها. |
| **B. Rules مفتوحة بأسماء Collections فقط** | لا تحل شيئًا: بدون Authentication حقيقية لا توجد قيمة موثوقة (`request.auth.uid`) لتُبنى عليها أي قاعدة تمييز بين المستخدمات. |
| **C. Firebase Anonymous Auth** | تعطي `uid` حقيقي وتُتيح قواعد أمان، لكن الحساب مرتبط بالمتصفح/الجهاز فقط — أي مسح للبيانات أو تبديل جهاز = فقدان الحساب بالكامل. غير مناسب لمشروع بيانات تراكمية طويلة المدى. |
| **D. Firebase Email/Password Auth** ✅ | **تم اختيار هذا الحل.** يعطي `uid` مستقر ومستقل عن الجهاز، ولا يحتاج Cloud Functions، ويعمل بالكامل من الواجهة الأمامية الثابتة الحالية. |
| **E. Firebase Auth + حسابات Admin وRoles** ✅ (مدمج مع D) | تم تطبيق الجزء الخاص بالـ Roles عبر مجموعة `roles/{uid}` بدل Custom Claims (لأن Custom Claims تحتاج استدعاء من بيئة موثوقة مثل Cloud Function أو سكربت Admin SDK، وهو تعقيد غير ضروري لعدد صغير من المشرفات). |
| **F. Backend/Cloud Functions للعمليات الحساسة** | الحل الأكمل نظريًا (خصوصًا لحماية اقتصاد اللعبة gems/XP بالكامل من التلاعب)، لكن يتطلب خطة Blaze المدفوعة وبنية خادم إضافية لا يحتاجها مشروع بهذا الحجم حاليًا. تم توضيح أين بالضبط لا تزال الحماية "نسبية" بسببه (القسم 9 أدناه)، بحيث يمكن إضافته لاحقًا دون إعادة هيكلة الحل الحالي. |

**السبب النهائي:** المشروع صغير (عدد محدود من المشاركات)، واجهة Static HTML/JS بدون خادم، ويحتاج فصلًا حقيقيًا بين "أنا" و"مشاركة أخرى" و"Admin" — وهذا بالضبط ما يوفره Firebase Email/Password Auth + قاعدة `roles/{uid}`، دون الحاجة لخطة مدفوعة أو خادم إضافي.

## 3) كيف يعمل تسجيل الدخول الآن؟

- كل مشاركة عندها بريد إلكتروني **اصطناعي** غير حقيقي، يُشتق رياضيًا من اسمها (SHA-256) — لا تحتاج المشاركة معرفته أو كتابته، الموقع يبنيه تلقائيًا في الخلفية. هذا فقط لأن Firebase Auth (Email/Password) يحتاج صيغة بريد صالحة كمعرّف دخول.
- شاشة الدخول الآن فيها تبويبان:
  - **"تسجيل الدخول"**: اسم + كلمة المرور → `signInWithEmailAndPassword`.
  - **"حساب جديد"**: اسم + كلمة مرور جديدة (6 أحرف فأكثر) + صورة رمزية → `createUserWithEmailAndPassword`.
- عند إنشاء حساب جديد بنفس اسم مشاركة قديمة، يبحث الموقع تلقائيًا عن تقدمها القديم (في `legacy-participants`) ويستعيده لها فور إنشاء الحساب (رسالة "تم استعادة تقدمك السابق 🎉").
- **كلمة المرور القديمة لا يمكن استيرادها تلقائيًا** (لأنها كانت SHA-256 من طرف العميل فقط، ولا توجد طريقة آمنة لتأكيد تطابقها مع تشفير Firebase الداخلي دون اختبارها فعليًا). لذلك كل مشاركة **تُنشئ كلمة مرور جديدة مرة واحدة فقط** عبر تبويب "حساب جديد"، وتقدمها القديم (done، الصورة، إلخ) يعود لها تلقائيًا. هذا ثمن ضروري وصريح لإصلاح ثغرة حقيقية، وتم إخباركِ به بدل الادّعاء بأنه غير موجود.

## 4) خطوات يدوية مطلوبة منكِ في Firebase Console

### أ) تفعيل Email/Password Authentication
1. افتحي [Firebase Console](https://console.firebase.google.com/) → مشروع **fitness-challenge-6061e**.
2. من القائمة الجانبية: **Build → Authentication → Get started** (أو Sign-in method إذا كانت مُفعّلة سابقًا).
3. فعّلي **Email/Password** فقط (لا حاجة لأي مزود آخر).
4. لا تحتاجين لإدخال أي مستخدمين يدويًا هنا — كل مشاركة تُنشئ حسابها من الموقع نفسه (تبويب "حساب جديد").

### ب) إنشاء أول حساب Admin
Firestore لا "تعرف" من هو Admin تلقائيًا. الطريقة:
1. افتحي الموقع، سجّلي دخولك بحسابك (أو أنشئي حسابًا جديدًا بنفس الطريقة العادية) — هذا يعطيكِ `uid` في Firebase Authentication.
2. من Firebase Console: **Authentication → Users**، انسخي الـ **User UID** الخاص بحسابك.
3. من **Firestore Database**: أنشئي وثيقة يدويًا:
   - Collection: `roles`
   - Document ID: **الصقي الـ UID الذي نسختِه**
   - أضيفي حقل: `admin` (Boolean) = `true`
4. سجّلي الخروج من الموقع ثم ادخلي مرة أخرى — الآن `admin.html` سيمنحك صلاحية الإدارة الحقيقية (وليس فقط شاشة كلمة المرور).

> يمكن تكرار هذه الخطوة لإضافة أكثر من Admin لاحقًا، فقط بإنشاء وثيقة `roles/{uid}` أخرى بنفس الطريقة.

### ج) نشر Firestore Rules
Firebase CLI **غير مثبّت في هذه البيئة** (تحققنا فعليًا: `firebase --version` غير موجود)، لذلك لم يتم نشر أي شيء تلقائيًا. نفّذي من جهازك:

```bash
npm install -g firebase-tools    # مرة واحدة فقط إذا لم يكن مثبتًا
firebase login
cd "مسار مجلد المشروع"
firebase deploy --only firestore:rules
```

ملف `.firebaserc` مضبوط مسبقًا على مشروع `fitness-challenge-6061e` (تحققي أن هذا هو مشروعك الصحيح قبل التنفيذ). الأمر أعلاه ينشر **فقط** ملف `firestore.rules` — لا ينشر Hosting ولا أي شيء آخر.

### د) تنفيذ Migration (استعادة الحسابات القديمة)

هذه الخطوة **اختيارية** فعليًا: أي مشاركة قديمة تستطيع ببساطة الدخول للموقع وإنشاء حساب جديد بنفس اسمها، وسيحاول الموقع تلقائيًا البحث عن `legacy-participants/{اسمها}` — لكنها لن تجد شيئًا ما لم يُشغَّل السكربت أولًا لنسخ بياناتها القديمة إلى هناك.

```bash
cd scripts
npm install
# احصلي على مفتاح Service Account من:
# Firebase Console → Project Settings → Service accounts → Generate new private key
# لا ترفعي هذا الملف إلى Git إطلاقًا (.gitignore يستثنيه مسبقًا)

# تجربة بدون كتابة فعلية:
GOOGLE_APPLICATION_CREDENTIALS=./service-account.json node migrate-users-to-auth.js --dry-run

# التنفيذ الفعلي:
GOOGLE_APPLICATION_CREDENTIALS=./service-account.json node migrate-users-to-auth.js
```

على PowerShell:
```powershell
$env:GOOGLE_APPLICATION_CREDENTIALS = "C:\path\to\service-account.json"
node migrate-users-to-auth.js --dry-run
```

السكربت:
- يحفظ **نسخة احتياطية JSON كاملة** لكل شيء في `scripts/backups/` قبل أي كتابة.
- لا يحذف ولا يعدّل وثائق `participants/` الأصلية أبدًا.
- ينسخ فقط (بدون `passwordHash`) إلى `legacy-participants/`.
- Idempotent: تشغيله عدة مرات آمن تمامًا.

## 5) كيف أتراجع إذا ظهرت مشكلة؟

- **القواعد:** إذا سببت `firestore.rules` الجديدة مشاكل غير متوقعة، يمكن العودة مؤقتًا لقواعد أكثر تساهلاً عبر `firebase deploy --only firestore:rules` بعد تعديل الملف - لكن **لا تعودي لـ `allow read, write: if true`** إطلاقًا، فذلك يعيد فتح كل الثغرات الموثقة في `FIREBASE_SECURITY_AUDIT.md`. إذا احتجتِ تخفيفًا مؤقتًا، افتحي فقط قراءة `exercises`/`public-profiles` (وهي مفتوحة أصلًا) ولا تفتحي `participants`/`game-state`/`notifications`.
- **البيانات:** كل الوثائق القديمة في `participants/` باقية دون حذف أو تعديل - لا يوجد أي سيناريو في هذا التنفيذ يحذف بيانات حقيقية.
- **الحسابات:** إذا أنشأت مشاركة حسابًا بالخطأ، يمكن حذفه من **Authentication → Users** في Console، ووثيقتها في `participants/{uid}` تُحذف يدويًا أو تبقى يتيمة بلا ضرر.

## 6) كيف أتحقق أن القواعد تعمل فعليًا؟

بعد النشر، من Firebase Console → Firestore → **Rules Playground** (أو من التطبيق مباشرة):
1. جرّبي قراءة `participants/{uid}` لمشاركة أخرى وأنتِ مسجّلة كمشاركة عادية → يجب أن تُرفض (Permission denied).
2. جرّبي تعديل `exercises/{id}` وأنتِ غير Admin → يجب أن تُرفض.
3. سجّلي دخولك كـ Admin وجرّبي إضافة تمرين من `admin.html` → يجب أن ينجح.
4. افتحي `leaderboard.html` بدون تسجيل دخول → يجب أن تظهر البيانات (لأنها تقرأ `public-profiles` العامة فقط) بدون أي `passwordHash` ظاهر في Network tab.

## 7) أين لا تزال الحماية "نسبية" (وتحتاج Cloud Function لاحقًا)؟

- **`game-state` (gems/bonusXp):** القواعد تمنع مشاركة من تعديل بيانات مشاركة أخرى، وتتحقق أن القيم أرقام غير سالبة، لكنها **لا تمنع** مشاركة من إرسال قيمة "gems: 999" لنفسها مباشرة عبر Console. الحل الكامل: نقل منطق "لف العجلة"/"فتح عنصر بالمتجر" إلى Cloud Function تتحقق من القواعد (مرة واحدة يوميًا، رصيد كافٍ...) وتكتب النتيجة بنفسها.
- **`publicStats` في `public-profiles`:** محسوبة ومكتوبة من طرف كل مشاركة عن نفسها. القواعد تمنع الكتابة لملف مشاركة أخرى، لكن لا تتحقق حسابيًا أن الرقم المُرسل صحيح فعلًا مقارنة بـ `done{}`. الحل الكامل: Cloud Function مُشغَّلة عند كل تعديل في `participants/{uid}` تعيد حساب `publicStats` من الخادم وتكتبها في `public-profiles`، بدل أن يحسبها العميل بنفسه.

هذا التنفيذ الحالي **أفضل بكثير** من الوضع السابق (حيث كان أي شيء ممكنًا للجميع)، لكنه ليس "صفر ثقة" 100% في هذه النقاط الضيقة تحديدًا - وهذا موثّق بصراحة بدل التستر عليه.
